# Orizum様向け インフルエンサーDB スプレッドシート定義書

**バージョン:** 1.0
**作成日:** 2026年1月20日
**作成者:** Manus AI

## 1. 概要

本ドキュメントは、Orizum社で運用されるインフルエンサーDBの基盤となるスプレッドシートの構造と運用ルールを定義するものです。現状の運用を整理し、将来的なWebアプリ化を見据えたデータ正規化と、日々の営業活動を効率化するビュー（表示形式）の設計を目的とします。

## 2. 全体方針

- **データの一元管理:** バイトによる情報追加、営業による進捗更新、システムによる自動更新の全てがこのスプレッドシート（将来的には裏側DB）に集約されます。
- **構造化と正規化:** 自由記述の「備考」欄を廃止し、連絡先や特徴などを個別の列に分離することで、検索性、自動処理、データ分析の精度を向上させます。
- **Webアプリへの移行性:** 本定義は、将来的にWebアプリケーションのデータベース（Creatorsテーブル）へスムーズに移行できることを前提に設計されています。

## 3. 列定義

スプレッドシートを4つの目的別グループに分けて列を定義します。

1.  **入力列 (カスタムGPT/バイト担当):** 主にカスタムGPTが出力し、バイトがペーストする情報。
2.  **営業進捗列 (営業担当):** 営業担当が日々の活動で更新する情報。
3.  **自動更新列 (システム担当):** YouTube APIなどを通じてシステムが定期的に更新する情報。
4.  **運用ログ列 (システム担当):** データがいつ、どのように更新されたかを記録する情報。

## 4. 営業向けビュー定義

営業担当者がスプレッドシートを開いた際に、その日にやるべきことが一目でわかるように、以下の4つのフィルタ済みビューを用意します。これにより、営業担当は各ビューを上から順に対応するだけで、効率的に日次業務を遂行できます。

| ビュー名 | 目的 | フィルタ条件 | ソート順 |
| :--- | :--- | :--- | :--- |
| **① 未DMリスト** | 新規に追加され、まだ誰もアプローチしていないインフルエンサーに初回DMを送る。 | `dm_status` = `未接触` | (任意) 登録者数の降順など |
| **② 追いDM対象リスト** | 前回のDMから一定期間が経過したインフルエンサーに追いDMを送る。 | `dm_status` = `初回DM済` OR `追いDM1済`<br>AND<br>`next_followup_at` <= `今日の日付` | `next_followup_at` の昇順 (期限が近い順) |
| **③ 返信ありリスト** | インフルエンサーから返信があったものに対応する。 | `dm_status` = `返信あり` | `last_contact_at` の降順 (最近返信があった順) |
| **④ 更新失敗リスト** | システムによる自動更新が失敗したレコードを確認し、手動で修正する。 | `update_status` = `failed` | `last_updated_at` の降順 |

## 5. 運用フロー

### 5.1. バイトの運用フロー

1.  カスタムGPTに必要な条件（ジャンル等）を入力し、インフルエンサー候補リストを出力させる。
2.  GPTの出力（`channel_url`, `account_name`, `handle`, `contact_type`, `contact_value`, `tags`）をコピーする。
3.  スプレッドシートの新しい行にペーストする。
    -   この時点で `dm_status` は自動的に `未接触` となるように設定することが望ましい。

### 5.2. 営業の運用フロー

1.  スプレッドシートを開き、「**① 未DMリスト**」ビューを確認し、担当者（`owner`）を自分に設定した上で初回DMを送信する。
2.  DM送信後、`dm_status` を `初回DM済` に更新し、`last_contact_at` を記録する。
3.  次に、「**② 追いDM対象リスト**」ビューを確認し、表示されているインフルエンサーに追いDMを送信する。
4.  送信後、`dm_status` を `追いDM1済` や `追いDM2済` に更新する。
5.  日々、「**③ 返信ありリスト**」を確認し、返信があった場合はステータスを `交渉中` などに変更し、対応する。
