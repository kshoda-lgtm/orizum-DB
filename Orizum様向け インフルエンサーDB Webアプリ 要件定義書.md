# Orizum様向け インフルエンサーDB Webアプリ 要件定義書

**バージョン:** 1.0
**作成日:** 2026年1月20日
**作成者:** Manus AI

## 1. 概要

本ドキュメントは、インフルエンサーDB運用プロジェクトの Phase 2「Webアプリ化」で開発するWebアプリケーションの要件を定義するものです。スプレッドシート運用で確立したデータ構造とワークフローを基盤とし、営業担当者の業務効率を最大化することを目的とします。

## 2. 目的とスコープ

### 2.1. 目的

-   **検索性の向上:** スプレッドシートでは困難だった、複数条件での柔軟なインフルエンサー検索を実現する。
-   **業務の可視化と効率化:** 「今日やるべきこと」をダッシュボードで明確にし、営業担当者が迷いなく日次業務を遂行できる環境を提供する。
-   **情報の一元化:** インフルエンサーの基本情報、指標、接触履歴、次のアクション、AIによるDM提案を1画面に集約し、意思決定を迅速化する。

### 2.2. スコープ

-   **対象ユーザー:** 主にOrizum社の営業担当者（5名）。
-   **対象外:** バイトによるインフルエンサーの新規追加プロセスは本アプリのスコープ外とし、引き続き「カスタムGPT → スプレッドシートへの直接入力」を前提とする。

## 3. ユーザー権限

初期リリースでは、2つのシンプルな権限グループを想定します。

| 権限グループ | 説明 | 主な操作 |
| :--- | :--- | :--- |
| **営業 (Sales)** | アプリケーションの主要ユーザー。自身の担当するインフルエンサーの管理と営業活動を行う。 | ・ダッシュボードの閲覧<br>・インフルエンサーの検索・閲覧<br>・担当インフルエンサーのステータス更新<br>・接触履歴の記録 |
| **管理者 (Admin)** | システム全体の管理を行う。営業担当者を兼任することも可能。 | ・全営業担当者の権限<br>・DMテンプレートの管理<br>・ユーザーアカウントの管理 |

## 4. 機能要件

### 4.1. 画面一覧

本アプリケーションは、主に以下の3つの画面で構成されます。

1.  **ダッシュボード (トップページ)**
2.  **検索・一覧画面**
3.  **クリエイター詳細画面**

### 4.2. 各画面の機能詳細

#### 4.2.1. ダッシュボード

営業担当者がログイン後に最初に表示される画面。「今日やるべきこと」をキュー形式で表示します。

-   **表示項目:**
    -   **未DMキュー:** `dm_status` が `未接触` のインフルエンサーリスト。
    -   **追いDM対象キュー:** `next_followup_at` が今日以前になっているインフルエンサーリスト。
    -   **返信ありキュー:** `dm_status` が `返信あり` のインフルエンサーリスト。
-   **フィルタリング:**
    -   担当者（`owner`）で絞り込みが可能。「自分担当」をデフォルト表示とする。
-   **アクション:**
    -   各リストのインフルエンサー名をクリックすると、クリエイター詳細画面に遷移する。

#### 4.2.2. 検索・一覧画面

DBに登録されている全インフルエンサーを検索・一覧表示する画面。

-   **検索条件:**
    -   キーワード検索（アカウント名、ハンドル、メモなど）
    -   特徴タグ（`tags`）
    -   登録者数（`subscriber_count`）の範囲指定
    -   連絡先タイプ（`contact_type`）
    -   DMステータス（`dm_status`）
    -   担当者（`owner`）
-   **ソート機能:**
    -   登録者数、最終更新日、次回フォロー日などで昇順・降順ソートが可能。
-   **表示形式:**
    -   テーブル形式で、主要な情報（アカウント名, 登録者数, ステータス, 担当者, 最終接触日）を表示。

#### 4.2.3. クリエイター詳細画面

特定のインフルエンサーに関する全ての情報を集約した画面。

-   **表示項目:**
    -   **基本情報:** アカウント名、ハンドル、チャンネルURL、連絡先情報、特徴タグ。
    -   **自動更新指標:** 登録者数、取得ジャンル、最終更新日時。
    -   **営業進捗:** 現在のDMステータス、担当者、次回フォロー日。
    -   **接触履歴タイムライン:** 過去のDM送信、返信内容のサマリーを時系列で表示。
    -   **AIによるDM提案:**
        -   状況（初回/追い/返信返し）に応じたDM文案を複数パターン提案。
        -   提案の根拠（例: 「このクリエイターは〇〇に興味があるため、この切り口を提案します」）も表示。
-   **アクション:**
    -   DMステータスの更新（例: 「初回DM済」ボタンをクリックするとステータスが変更される）。
    -   接触履歴の新規追加（送信したDMの内容などを記録）。
    -   メモの追記・編集。

## 5. データ同期方針

-   **初期段階:** Webアプリは、既存のスプレッドシートをデータベースとして直接参照・更新します（Google Sheets API経由）。これにより、バイトの運用フローを変更することなく、スムーズにWebアプリを導入できます。
-   **将来的な移行:** アプリケーションのパフォーマンスやデータ量の増加に伴い、スプレッドシートから本格的なリレーショナルデータベース（MySQL, PostgreSQLなど）への移行を検討します。その際は、データ移行用のスクリプトを別途開発します。

## 6. 非機能要件

| 項目 | 要件 |
| :--- | :--- |
| **パフォーマンス** | 各画面の表示は3秒以内を目標とする。 |
| **セキュリティ** | ログイン認証を必須とし、パスワードはハッシュ化して保存する。 |
| **可用性** | 99.5%以上の稼働率を目指す。 |
| **ブラウザ** | 最新版のGoogle Chrome, Firefox, Safariに対応する。 |
